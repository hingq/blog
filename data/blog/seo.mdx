# 网站收录

很多个人网站在上线之后都能正常访问，但过了几天甚至几周，搜索引擎里还是搜不到。

问题通常不在“网站能不能打开”，而在于搜索引擎能不能稳定地理解下面几件事：

1. 这个站点的正式域名到底是哪一个。
2. 哪些页面允许抓取、允许索引。
3. 站内是否提供了明确的页面清单。
4. 页面正文是不是足够清晰、足够有价值。

如果这几件事没有处理好，搜索引擎就算偶尔爬到了页面，也可能不会收录，或者会把错误的地址当成正式地址。

这篇文章按实际操作顺序整理一遍，从技术准备到提交站长平台，再到最后的排查清单，做完这一套，基础收录问题通常就能解决。

## 第一步：先确认收录前置条件

在看 `robots.txt`、`sitemap.xml` 之前，先确认几个更基础的问题。

### 1. 网站必须能被公开访问

如果页面需要登录、验证码、白名单，或者地区网络访问不稳定，搜索引擎抓取就会受影响。

最低要求是：

1. 首页能直接访问。
2. 文章详情页能直接访问。
3. 页面正文不依赖登录后接口才出现。

如果正文完全依赖浏览器端请求接口再渲染，抓取成功率会明显下降。对博客类网站来说，正文最好直接出现在服务端返回的 HTML 中。

### 2. 只能有一个主域名

这是最容易被忽略的问题。

例如下面几个地址，从用户角度看可能是同一个网站：

- `http://fortunately.top`
- `https://fortunately.top`
- `https://www.fortunately.top`
- `https://your-project.vercel.app`

但从搜索引擎角度看，它们是不同 URL。如果这些地址都能打开同样的内容，就会出现重复页面、canonical 混乱、权重分散的问题。

正确做法是只保留一个正式域名，其他地址全部 `301` 到它。对大多数个人站来说，正式域名应该是：

```text
https://你的域名
```

### 3. 页面不能显式禁止索引

要确认页面里没有这些问题：

- `meta name="robots" content="noindex"`
- HTTP Header 返回了 `X-Robots-Tag: noindex`
- `robots.txt` 错误屏蔽了正文路径

如果页面被标记为 `noindex`，你做再多 sitemap 提交也没有意义。

### 4. 网站不能只有空壳

新站最常见的误区是：技术配置都做了，但站内只有一篇很短的文章，或者内容高度重复。

搜索引擎会评估页面质量。至少要保证：

1. 首页有明确导航。
2. 列表页可以进入详情页。
3. 文章页正文足够完整，不是只有一两句话。
4. 站内页面之间存在自然内链。

## 第二步：准备好收录所需的技术信号

搜索引擎最关心的是“这个站点的正式地址是什么”和“有哪些页面值得抓取”。

### 1. 配置统一的站点主域名

无论你使用什么博客框架，都应该有一个统一的站点域名配置。

以 Next.js 博客为例，通常会有一个类似 `siteUrl` 的配置：

```js
const siteMetadata = {
  siteUrl: 'https://fortunately.top',
}
```

这个值会进一步影响：

- canonical
- sitemap
- robots
- Open Graph URL
- RSS 链接

如果这里写成了模板默认域名，或者写成了临时预览域名，搜索引擎就会被你错误地引导到别的地址。

### 2. 提供 robots.txt

`robots.txt` 的作用不是“让搜索引擎一定收录”，而是明确告诉它哪些路径可以抓，sitemap 在哪里。

一个最基础的版本通常像这样：

```txt
User-agent: *
Allow: /

Sitemap: https://fortunately.top/sitemap.xml
Host: https://fortunately.top
```

关键点有两个：

1. 不要误把博客正文目录屏蔽掉。
2. 确保 `Sitemap` 指向正式域名下的 sitemap。

### 3. 提供 sitemap.xml

`sitemap.xml` 就是你主动递给搜索引擎的“页面目录”。

至少应该包含：

- 首页
- 博客列表页
- 标签页或分类页
- 所有公开文章页

如果你用 Next.js App Router，通常会通过 `app/sitemap.ts` 动态生成。核心要求不是实现方式，而是输出结果正确：

1. 所有 URL 都是正式域名。
2. 不包含草稿页面。
3. 不包含 `404`、测试页、预览页。
4. `lastModified` 尽量准确。

### 4. 给页面设置 canonical

canonical 的作用是告诉搜索引擎：“如果你发现多个相似地址，请把这个地址当成正式版本。”

对于博客站，canonical 尤其重要，因为同一篇文章可能会通过不同方式被访问：

- 带参数 URL
- 分页入口
- 预览域名
- `http` 和 `https`

页面里最好稳定输出 canonical，例如：

```tsx
alternates: {
  canonical: './',
}
```

只要 `metadataBase` 或站点主域名配置正确，最终生成的 canonical 就会落到正式域名上。

### 5. 基础 metadata 要完整

收录不完全等于 SEO，但最基础的页面信息还是要有。

至少建议保证这些字段存在：

- `title`
- `description`
- canonical
- Open Graph title / description / image

这些字段主要有两个作用：

1. 帮助搜索引擎理解页面主题。
2. 让页面在分享和搜索结果中展示得更稳定。

## 第三步：把站点提交给 Google 和 Bing

技术信号准备好之后，不要等搜索引擎“自己发现”，直接提交会更快。

### 1. Google Search Console

流程很简单：

1. 打开 Google Search Console。
2. 添加站点。
3. 完成域名或 URL 前缀验证。
4. 提交 `https://你的域名/sitemap.xml`。
5. 对首页和重要文章页使用“网址检查”请求抓取。

验证方式里，域名验证通常更彻底；如果你暂时不方便改 DNS，也可以先使用 URL 前缀验证。

提交之后重点看两个地方：

- `Sitemaps` 是否读取成功
- `Pages` 或覆盖率报告里是否出现 “Alternate page with proper canonical”、`noindex`、`Crawled - currently not indexed` 等提示

### 2. Bing Webmaster Tools

Bing 的流程和 Google 基本一致：

1. 添加站点。
2. 验证所有权。
3. 提交 `sitemap.xml`。
4. 查看抓取和索引状态。

很多时候 Bing 还可以直接导入 Search Console 的站点配置，操作成本更低。

如果你的目标用户并不只来自 Google，Bing 这一步不要省略。

## 第四步：为什么提交了 sitemap，还是不收录

这是最常见的问题。`sitemap.xml` 只是“告诉搜索引擎页面存在”，不是“强制要求它收录”。

下面这些原因都很常见。

### 1. 新站时间太短

新域名、新网站、新内容，搜索引擎通常需要时间建立信任。

如果技术配置已经正确，刚提交两三天还没有明显收录，不一定是配置错了，也可能只是还在排队。

### 2. 页面价值太弱

下面这些内容往往不容易被收录：

- 标题和正文严重不匹配
- 文章极短，信息密度低
- 同质化内容过多
- AI 批量生成但缺乏实际信息

搜索引擎真正想收录的是“值得被搜索到”的页面，而不是“存在一个 URL”。

### 3. 重复页面太多

常见来源包括：

- `http` 和 `https` 同时可访问
- 主域名和预览域名同时开放
- 带尾斜杠和不带尾斜杠策略不一致
- 同一内容在多个栏目路径下重复出现

如果重复页面太多，搜索引擎会更谨慎，甚至优先不收录。

### 4. 页面抓得到，但暂时不想收

在 Search Console 里，`Crawled - currently not indexed` 很常见。它的意思不是抓取失败，而是抓到了，但暂时没决定收录。

这时候要优先看：

1. 内容是否足够有价值。
2. 页面是否与站内其他页面重复。
3. canonical 是否指向了别处。
4. 站点整体质量和更新频率是否过低。

## 排查清单

### 1. 检查 robots.txt

直接打开：

```text
https://你的域名/robots.txt
```

确认：

- 能访问
- 没有屏蔽正文目录
- 正确声明 `sitemap.xml`

### 2. 检查 sitemap.xml

直接打开：

```text
https://你的域名/sitemap.xml
```

确认：

- 能访问
- URL 都是正式域名
- 没有草稿和测试页面

### 3. 检查跳转是否统一

用命令看响应头最直接：

```bash
curl -I http://你的域名
curl -I https://你的域名
```

你要看到的是：

1. 非正式域名跳到正式域名。
2. 最终落点是 `https://`。
3. 不要出现多次来回跳转。

### 4. 检查页面源码

打开一篇文章页的“查看源代码”，确认：

- 有 `title`
- 有 `description`
- 有 canonical
- 没有 `noindex`

如果你用的是 React 或 Next.js，不要只看浏览器渲染后的 DOM，要看最初返回的 HTML。

### 5. 检查站长平台报告

最后再去 Google Search Console 和 Bing Webmaster Tools 看后台报告。

重点不是“有没有马上收录”，而是有没有明显错误：

- sitemap 读取失败
- 页面被 `noindex`
- canonical 指向其他域名
- 页面被重定向
- 抓取异常
