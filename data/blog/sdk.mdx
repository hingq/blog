做前端埋点时，最常见的两个指标是 PV 和 UV。PV 是页面被访问的次数，UV 是去重后的独立访客数。它们听上去简单，但一旦落到真实项目里，页面形态、路由模式、浏览器能力和页面卸载时机都会直接影响采集结果。

如果要实现一个最小可用的前端埋点 SDK，核心问题其实只有两个：什么时候采集，以及怎样把数据尽量稳定地发出去。本文就围绕这两个问题，梳理一套从采集到上报的基础设计。

## 核心流程：从校验到采集，再到上报

一个埋点 SDK 的主链路通常可以拆成三个阶段：`Storage`、`Collection` 和 `Reporting`。

`Storage` 负责识别用户身份与访问状态。最常见的做法是在初始化时检查 `localStorage`、`cookie` 或其他持久化存储，判断当前用户是首次访问还是回访用户。对 UV 来说，这一步的作用是做客户端去重，避免同一用户在短时间内被重复统计成多个独立访客。

`Collection` 负责监听行为事件并组装原始数据。这里至少要覆盖页面浏览，也就是 PV；如果业务需要更细的分析，还可以继续采集点击事件、页面停留时长、来源页面和自定义业务事件。埋点 SDK 真正的价值，不是把所有事件都监听一遍，而是在合适的时机捕获关键行为。

`Reporting` 负责把采集到的数据发送给服务端。这个阶段除了“发出去”以外，还要考虑队列聚合、失败兜底、页面关闭时的丢包风险，以及不同浏览器对发送方式的支持差异。

## 页面浏览采集：MPA 和 SPA 的思路完全不同

PV 看似只是“打开一次页面就记一次”，但不同站点模式下，触发时机并不一样。

对于传统 MPA 页面，每次跳转都会触发一次完整加载，因此在 `window.load` 或更早的初始化阶段采集 PV 通常就够用了。除此之外，最好同时记录 `document.referrer`，这样服务端才能判断用户是从搜索、外链还是站内页面跳转而来。

对于 SPA，页面不会整体刷新，路由变化更多是前端层面的 URL 变更。只监听 `window.load` 会漏掉绝大多数内部跳转，因此需要补上对 History API 和浏览器导航行为的监听。

常见的三类路由变化分别是：

1. 代码触发跳转，例如 `router.push`、`router.replace`，本质上会调用 `history.pushState` 和 `history.replaceState`。
2. 用户点击浏览器前进或后退按钮，对应 `popstate` 事件。
3. 使用 hash 路由时，URL 中 `#` 后面的内容变化，对应 `hashchange` 事件。

其中第一类通常需要通过重写原生方法来拦截：

```ts
const originalPush = history.pushState
const originalReplace = history.replaceState

history.pushState = function (...args: Parameters<typeof history.pushState>) {
  originalPush.apply(this, args)
  handleRouteChange()
}

history.replaceState = function (...args: Parameters<typeof history.replaceState>) {
  originalReplace.apply(this, args)
  handleRouteChange()
}
```

这段代码的重点不是“重写方法”本身，而是在路由真正变更完成后，补一次统一的 `handleRouteChange` 处理。这个处理函数里通常会做三件事：读取当前 URL、生成新的 PV 事件、附带来源信息后进入上报队列。

如果项目同时支持 `popstate` 和 `hashchange`，还需要把这些入口统一收敛到同一套路由变更逻辑里，否则很容易出现某些跳转能统计、某些跳转漏记的情况。

## 除了 PV，还可以补上点击与停留时长

在很多业务里，PV 只能告诉你“页面被看到了”，但无法说明“用户在页面里做了什么”。因此一个更完整的埋点 SDK，通常还会继续采集点击事件和停留时长。

点击事件适合做交互路径分析，例如按钮点击、模块曝光后的转化动作等。实现时要注意不要把所有点击都无差别上报，而是尽量基于元素标识、业务属性或手动埋点配置去筛选目标节点，否则上报量和噪声都会很高。

停留时长则更适合衡量内容消费质量。最基础的做法是记录进入页面的时间戳，在页面切换、隐藏或卸载时计算停留时间并上报。这里要特别注意，用户切到后台标签页并不等于真正离开页面，因此在工程上通常还会结合 `visibilitychange` 一起判断，避免停留时长被高估。

## 数据上报：优先 Beacon，其次 keepalive，最后才是图片打点

浏览器环境里最容易丢数据的时机，就是页面关闭、刷新或跳转离开的时候。因此上报方案不能只看“能不能发请求”，还要看页面生命周期结束时能不能尽量发完。

第一选择通常是 `navigator.sendBeacon`。它本来就是为页面卸载场景设计的，浏览器会在后台继续发送数据，不会阻塞页面关闭，对统计类请求非常友好。如果只是发送轻量埋点数据，而且不需要复杂请求头，Beacon 往往是最稳妥的方案。

第二选择是 `fetch` 加 `keepalive: true`。当你需要自定义 Header、统一走现有请求封装，或者浏览器环境不适合使用 Beacon 时，可以用这种方式兜底。它的核心价值在于告诉浏览器：“即使页面即将卸载，这个请求也尽量帮我发完。”

第三选择才是图片打点，也就是创建一个 `img`，把参数拼到 URL 上发 GET 请求。这种方式历史最久，兼容性很好，而且不依赖跨域读取能力。但它只能做 GET，请求长度有限，也不适合承载复杂 payload，更适合作为极简兜底方案存在。

如果把这三种方式放在一个 SDK 里，最常见的策略就是按能力降级：优先尝试 Beacon，不满足条件时退到 `fetch + keepalive`，再不行就回退到图片打点。

## 一个最小可用的前端埋点 SDK，至少要解决什么问题

总结下来，一个最小可用的前端埋点 SDK 不需要一开始就做得很大，但至少要把以下能力打通：

1. 能识别用户访问状态，支撑 UV 去重。
2. 能在 MPA 和 SPA 场景下正确采集 PV。
3. 能扩展点击、停留时长这类常见行为事件。
4. 能在页面卸载时尽量可靠地把数据发送出去。
5. 能对不同浏览器能力做兼容和降级。

把这些基础能力做好以后，后续无论是接入更多业务事件、增加批量上报、补充错误重试，还是接入更完整的数据分析平台，都会顺畅得多。埋点 SDK 的难点从来不只是“发一个请求”，而是用尽可能稳定的方式，把浏览器里真正有价值的行为信号采集完整。
